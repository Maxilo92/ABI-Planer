# ABI Planer v1.0.0 Readiness Review
**Date**: 2026-03-28
**Scope**: TCG Logic, Internal Security, Legal/Tax Compliance, Operations

## Executive Summary
The "ABI Planer" application has been rigorously audited across its core subsystems. The architecture is sophisticated and highly capable, particularly in handling delayed administrative tasks ("Danger System") and frontend consistency.

However, the application is **NOT READY** for a public v1.0.0 release. We have identified **Critical blockers** in TCG Logic and Firestore Security that could lead to immediate exploitation or data exposure. The Legal/Tax compliance is strong for digital goods but requires minor adjustments for GDPR completeness.

---

## 1. Critical Blockers (Must Fix Before v1.0.0)

### 1.1 Client-Side RNG Vulnerability (TCG Logic)
* **Risk**: High (Exploitable)
* **Finding**: The card generation logic (`generatePack`, `getRandomVariant`) is executed in the browser (`src/app/sammelkarten/page.tsx`). A malicious user can intercept or override these functions to guarantee the drop of "Legendary" or "Secret Rare" cards, bypassing all intended probabilities and daily limits.
* **Resolution**: Move all booster opening and card variant calculation logic to a secure Cloud Function. The client should only trigger the `openBooster()` endpoint.

### 1.2 Public Read Access (Security)
* **Risk**: High (Data Exposure)
* **Finding**: The `firestore.rules` file allows public internet access (`allow read: if true;`) to several critical collections including `/news`, `/events`, `/finances`, `/polls`, and `/teachers`. This violates the intended "Zero Trust" architecture.
* **Resolution**: Replace all instances of `allow read: if true;` with `allow read: if isAuthenticated();` (or `isApproved()`) to ensure only authorized users within the `@hgr-web.lernsax.de` domain can read the data.

### 1.3 Permissive Write Access (Security)
* **Risk**: High (Exploitable)
* **Finding**: The `firestore.rules` allow users to freely modify their own `user_teachers` document (`allow write: if isAuthenticated() && request.auth.uid == userId;`).
* **Resolution**: Restrict write access. Changes to `user_teachers` should only occur server-side via the Admin SDK after a valid booster purchase or opening.

### 1.4 Incomplete GDPR Deletion (Compliance)
* **Risk**: High (Legal Liability)
* **Finding**: The `onProfileDeleted` Cloud Function cleans up user data but fails to delete records in the `referrals` collection (`std_${userId}`), leaving Personally Identifiable Information (PII) intact.
* **Resolution**: Add the `referrals` collection to the deletion sequence in `functions/src/users.ts`.

## 2. Major Recommendations (Should Fix Before v1.0.0)

### 2.1 Full Rarity Synchronization
* **Issue**: The `calculateTeacherRarity` Cloud Function only updates the specific teacher that triggered the recalculation. If rankings shift (e.g., a teacher loses the "Legendary" spot due to global limits), the displaced teacher's Firestore document is not immediately updated.
* **Resolution**: Implement a periodic sync (Cron job) or a more comprehensive transaction to update all affected teachers simultaneously.

### 2.2 Explicit Billing Address Collection for Stripe
* **Issue**: For robust MOSS (VAT on Digital Services) compliance in the EU, merchants must collect non-conflicting location evidence.
* **Resolution**: Update the Stripe Checkout Session configuration in `functions/src/shop.ts` to enforce `billing_address_collection: "required"`.

---

## 3. Minor Recommendations (Post-v1.0.0 Polish)

### 3.1 CI/CD Improvements
* **Issue**: The current GitHub Actions workflow (`regression-guard.yml`) relies heavily on static regex analysis. It does not run existing unit tests (`functions/__tests__/rarity.test.ts`) or linting.
* **Resolution**: Integrate `npm run lint` and a test runner into the pipeline to prevent logic regressions.

### 3.2 Audit Log Anonymization
* **Issue**: The `logs` collection retains `user_id` references indefinitely.
* **Resolution**: Implement a strategy to scrub or anonymize the `user_id` from logs when a user account is deleted, balancing security auditing needs with GDPR's right to be forgotten.

---

## Final GO/NO-GO Decision
**NO-GO**. The application requires immediate remediation of the client-side TCG RNG and Firestore security rules before any public exposure. Once the 4 Critical Blockers are addressed, the application will be cleared for launch.
